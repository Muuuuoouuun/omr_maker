# OMR Maker 서비스 방향 정리

작성일: 2026-06-16

## 확정 방향

- 문항 DB는 당장 문항 이미지를 별도 저장하지 않고, 원본 PDF와 `pdf_region` 좌표, canonical question ID, 유형 태그, 정답/배점으로 먼저 운영한다.
- 프리미어 단계에서는 PDF 영역을 문항별 이미지 asset으로 커팅해 저장하고, 장기적으로 문항 DB를 유형/난이도/출처/오답률 기준으로 고도화한다.
- 감독자는 실시간 제어보다 결과 확인이 우선이다. 강제 종료, 응시/미응시 확인, 결과 요약까지를 먼저 안정화하고, 고급 감독 기능은 뒤로 둔다.
- 알림/초대의 1차 운영 채널은 카카오다. 현재 제품 화면에서는 실제 발송처럼 보이지 않게 “연동 전/기록” 상태로 표시한다.
- 결제 연동 순서는 토스페이먼츠, 네이버페이, 카카오페이다. 지금은 로컬 플랜 기록만 제공하고 실제 과금은 하지 않는다.
- 학생 계정도 제품 범위에 포함한다. 게스트 풀이 기록을 학생 계정으로 합치는 흐름을 Supabase Auth 전환과 함께 설계한다.
- 분석의 1차 운영 축은 지역 기반이다. 이후 학생별, 반별, 시험별, 문항별, 유형별 축을 모두 제공하되, 중간 수준까지 정확히 작동하는 것을 먼저 목표로 한다.
- 반 이름이 같아도 지역이 다르면 별도 운영 단위로 본다. 학생/반/시험 분석과 카카오 후보, 배포 대상 계산은 지역 스냅샷을 우선 반영한다.

## 구현 우선순위

1. 시험 제작/배포: OMR 기본값, PDF/정답 인식, 배포 링크, 접근 제어.
2. 학생 풀이/필기 UX: 태블릿 PDF 필기, 오른쪽 접이식 OMR 사이드바, 자동 저장, 제출 복구.
3. 관리자 분석: 틀린 문제, 유형, 학생별/반별/시험별 분석, 재추천 후보.
4. 결제/플랜/인증: Supabase Auth, 학생 계정, 카카오 알림, 결제 연동.

## 문항 이미지 저장 여부

초기 운영에는 문항 이미지를 저장하지 않아도 된다. 틀린 문제/유형 분석에 필요한 최소 단위는 다음이다.

- 시험 ID
- 문항 번호와 canonical question ID
- 정답, 학생 답, 정오답, 배점
- 유형 태그와 단원 태그
- PDF page/x/y/width/height 좌표

이 구조면 “학생별”, “반별”, “시험별”, “유형별” 분석은 가능하다. 다만 학생에게 틀린 문항을 시각적으로 다시 보여주거나 프리미어 문항 DB를 만들려면, 같은 좌표에서 원본 PDF를 커팅해 이미지 asset을 저장하는 단계가 필요하다.

## 추천 로직 단계

1. 약점 표시: 학생/반/시험 기준으로 오답 문항과 유형을 정확히 묶는다.
2. 추천 액션: 많이 틀린 유형, 최근 반복 오답, 반 평균 대비 취약 영역을 기준으로 복습 후보를 제안한다.
3. 고급 개인화: 문항 이미지 DB, 장기 이력, 난이도, 유사 문항 매칭을 붙여 프리미어 추천으로 확장한다.

현재 목표는 1~2단계를 제대로 만드는 것이다.

## 확정된 운영 응답 반영

- 지역 기반을 우선 축으로 둔다.
- 감독자는 결과 확인 중심으로 설계한다.
- 알림은 카카오를 1차 채널로 둔다.
- 결제는 토스페이먼츠, 네이버페이, 카카오페이 순서로 붙인다.
- 학생 계정도 포함한다.
- 기능 범위는 전부 가져가되, 고급 기능보다 중간 수준의 안정성을 먼저 만든다.
- 1~2단계 추천 로직을 우선 완성하고, 이후 문항 이미지 DB 기반 고급 개인화로 확장한다.
