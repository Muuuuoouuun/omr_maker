# 국어 실전 시험 Supabase 샘플 배포 설계

## 목표

사용자가 제공한 세 개의 한국교육과정평가원 국어 문제지로 실제 앱 흐름을 검증할 수 있는 샘플 시험 묶음을 만든다. 연결된 Supabase의 `teacher_sharedqa` 조직과 `테스트반`에 시험을 배포하고, 학생별로 미응시·오답 리뷰·오답 재시험 상태를 준비한다. 교사는 시험 제작 결과, 배포 상태, 응시 결과, 필기 보관, 피드백, 오답 분석을 실제 화면에서 확인할 수 있어야 한다.

샘플 등록은 QA 전용이며 운영 사용자나 실제 학생 데이터를 대상으로 하지 않는다. 기존 미커밋 대시보드 변경과 공개 `omr-showcase` 목업 데이터는 수정하지 않는다.

## 원본 시험과 정규화

| 샘플 ID | 원본 | 처리 | 앱 표시 이름 |
|---|---|---|---|
| `sample-korean-2025-csat-media` | 2025학년도 대학수학능력시험 국어 영역 홀수형, 20쪽 | 공통 + 언어와 매체 범위로 정규화 | `[샘플] 2025학년도 수능 국어 - 언어와 매체` |
| `sample-korean-2026-september-media` | 2026학년도 9월 모의평가 국어, 20쪽 | 공통 + 언어와 매체 범위로 정규화 | `[샘플] 2026학년도 9월 모평 국어 - 언어와 매체` |
| `sample-korean-2026-csat-media` | 2026학년도 대학수학능력시험 국어, 홀수형·짝수형 합본 40쪽 | 앞 20쪽 홀수형에서 공통 + 언어와 매체 범위만 사용 | `[샘플] 2026학년도 수능 국어 - 언어와 매체` |

각 시험은 1~34번 공통 문항과 35~45번 언어와 매체 문항으로 구성한다. 화법과 작문 페이지와 세 번째 원본의 짝수형 페이지는 학생용 정규화 PDF에서 제외한다. 원본 파일은 변경하지 않고, 정규화 PDF만 저장소의 임시 산출물로 만든 뒤 Supabase Storage에 올린다.

정답과 배점은 한국교육과정평가원의 공식 정답 자료를 우선 사용한다. 공식 자료의 시험명·시행일·홀수형·선택과목을 샘플과 대조하고, 45문항의 정답 범위가 1~5인지와 총점이 100점인지 등록 전에 검증한다. 공식 자료를 확인하지 못한 시험은 추정값으로 등록하지 않고 전체 적용을 중단한다.

## 시험 모델

세 시험 모두 다음 조건을 사용한다.

- 조직: `teacher_sharedqa`
- 생성 교사: 공유 QA 관리자 계정의 actor ID
- 배포 대상: `테스트반`의 고정 class/group ID
- 접근 방식: `accessConfig.type = "group"`
- 문항 수: 45문항, 5지선다
- 시험 시간: 80분
- 공개 기간: 적용 시점부터 30일
- 보관 상태: 활성
- 문제 PDF: Supabase 비공개 버킷의 `problem_pdf` 원격 자산
- 과목 태그: `국어`
- 출처 태그: 각 시험의 공식 시행명
- 영역 태그: 문항 범위에 따라 독서·문학·언어와 매체 등 확인 가능한 수준으로 부여

문항별 정답, 배점, PDF 페이지 위치를 canonical exam payload와 `omr_exam_questions` 양쪽에 동일하게 저장한다. 자동 위치 감지가 확실하지 않은 문항은 잘못된 위치를 저장하지 않고 위치 메타데이터를 비워 둔다. 정답과 채점 가능 여부는 PDF 위치 인식과 독립적으로 보장한다.

현재 학생 배포 화면은 `omr_assignments`보다 시험 payload의 그룹 접근 설정을 기준으로 동작한다. 따라서 이번 범위에서는 앱이 실제로 사용하는 그룹 배포 경로를 권위값으로 삼고, 사용되지 않는 assignment 행을 별도로 만들지 않는다.

## 학생별 시나리오

### student1: 전체 흐름 완료

첫 번째 시험에 원응시와 오답 재시험을 등록한다.

- 원응시에는 정답·오답·미응답이 섞인 결정적 답안과 문항별 시간 기록을 저장한다.
- PDF 필기 JSON을 비공개 Storage에 저장하고 원응시에 `drawingsRef`와 필기 요약을 연결한다.
- 교사 피드백은 `returned` 상태로 저장하며, 요약과 일부 오답 문항 코멘트를 포함한다.
- 원응시 오답만 대상으로 재시험 응시를 등록하고, 일부 문항은 회복되고 일부는 다시 틀리게 구성한다.
- 재시험은 원응시 ID, `wrong` 모드, 대상 문항 ID를 보존하여 학생 리뷰의 회복률 카드가 동작하게 한다.

### student2: 리뷰 후 재시험 대기

첫 번째 시험의 원응시만 등록한다.

- 오답이 존재하고 채점 결과가 완료 상태여야 한다.
- 학생 리뷰에서 오답 필터와 `오답만` 재시험 링크가 활성화되어야 한다.
- 재시험 응시는 미리 만들지 않아 사용자가 직접 흐름을 실행할 수 있게 한다.

### student3: 신규 응시

세 시험 모두 미응시 상태로 둔다. 학생 대시보드에서 배포된 시험을 열고 OMR 답안과 PDF 필기를 직접 제출할 수 있어야 한다.

두 번째와 세 번째 시험은 모든 학생에게 미응시 시험으로 남겨 원본별 PDF 열기와 신규 응시를 반복 검증할 수 있게 한다.

## Supabase 저장 구조

멱등 등록 스크립트는 Vercel Production 환경에서 Supabase URL과 서비스 역할 키를 임시로 읽고 다음 순서로 작업한다.

1. 대상 조직·테스트반·학생 3명이 기존 fixture와 일치하는지 확인한다.
2. 정규화 PDF와 필기 JSON의 해시를 계산한다.
3. `omr-private-assets` 버킷의 고정 조직 경로에 자산을 upsert한다.
4. `omr_remote_assets` 메타데이터를 자산 내용과 일치시킨다.
5. 기존 canonical 변환 규약을 사용해 `omr_exams`와 `omr_exam_questions`를 upsert한다.
6. 고정 attempt ID로 `omr_attempts`와 `omr_question_results`를 upsert한다.
7. student1의 반환 피드백과 필기 연결을 upsert한다.
8. 전체 읽기 검증을 통과한 뒤 완료를 보고한다.

시험, 자산, 응시, 피드백은 모두 고정 ID를 사용한다. 같은 파일과 같은 fixture로 재실행하면 행이나 Storage 객체가 중복되지 않는다. 입력 파일 해시 또는 예상 시험 구조가 바뀐 경우에는 검증 오류를 내고, 명시적 갱신 로직이 없는 상태에서 조용히 덮어쓰지 않는다.

## 구현 구성

- 순수 fixture 빌더: 시험 사양, 문항, 응시, 재시험, 피드백을 생성한다.
- PDF 정규화 단계: 선택한 페이지만 새 PDF로 묶고 페이지 수와 첫/마지막 페이지 표식을 검증한다.
- Supabase 적용 단계: Vercel 환경을 임시 디렉터리에 가져오고 자산·행을 멱등 반영한다.
- 검증 단계: DB 행, 원격 자산, 학생 접근, 정답 총점, 재시험 연결을 읽기 전용으로 확인한다.
- 평가 문서: 제작·배포·응시·필기·리뷰·재시험 요소별 결과와 수동 확인 링크를 기록한다.

스크립트는 `--dry-run`, `--apply`, `--verify` 모드를 제공한다. `--dry-run`과 `--verify`는 Supabase를 변경하지 않는다. 로그에는 서비스 역할 키, Vercel 토큰, 세션 비밀값, 학생 시작 코드 해시를 출력하지 않는다.

## 오류 처리와 복구

- 대상 조직·반·학생 fixture가 다르면 적용하지 않는다.
- 공식 정답 검증, 45문항 검증, 100점 검증, PDF 페이지 검증 중 하나라도 실패하면 원격 변경 전에 중단한다.
- Storage 업로드 후 DB 저장이 실패하면 새로 만든 해당 실행의 자산을 정리하거나, 고정 ID 자산을 다음 실행에서 안전하게 재사용할 수 있는 상태로 둔다.
- DB 적용은 외래 키 순서에 맞추고, 검증 실패 시 어떤 ID가 불일치하는지 비밀값 없이 표시한다.
- 기존 동일 ID의 데이터가 fixture 소유 표식을 갖지 않으면 덮어쓰지 않는다.
- 부분 적용 후 재실행해도 누락 단계만 복구할 수 있어야 한다.

샘플 제거가 필요할 경우 고정 ID 목록으로 시험·응시·피드백·원격 자산만 제거할 수 있는 별도 `--remove` 모드를 제공한다. 제거는 명시적으로 실행할 때만 수행하고 테스트반·학생 계정은 삭제하지 않는다.

## 테스트와 검증

자동 테스트는 다음을 확인한다.

- 세 시험이 각각 45문항이고 정답이 1~5이며 총점이 100점이다.
- 시험 payload와 question row의 정답·배점·선택지 수가 일치한다.
- PDF 정규화 결과에 짝수형과 화법과 작문 범위가 포함되지 않는다.
- 그룹 접근 설정이 `테스트반`만 대상으로 한다.
- student1 원응시의 오답 ID와 재시험 대상 ID가 일치한다.
- student1 재시험 결과로 회복·재오답이 모두 계산된다.
- student2에는 원응시만 있고 재시험 링크를 만들 수 있다.
- student3에는 사전 생성 응시가 없다.
- 필기 원격 자산의 조직·attempt scope와 해시가 일치한다.
- 같은 fixture를 두 번 적용해도 행과 객체 수가 증가하지 않는다.

적용 후에는 다음을 실제 배포 앱에서 확인한다.

1. 교사 계정에서 세 시험과 `테스트반` 배포 상태를 조회한다.
2. student1이 원응시 리뷰, 반환 피드백, 필기 상태, 재시험 회복률을 본다.
3. student2가 오답 리뷰에서 재시험을 시작한다.
4. student3가 시험을 열고 PDF에 필기한 뒤 제출한다.
5. 교사 화면에서 신규 제출과 오답·문항 분석이 갱신된다.

코드 검증은 순수 빌더 단위 테스트, 관련 Supabase 계약 테스트, 전체 Vitest, lint, production build 순서로 수행한다. 외부 적용 뒤에는 `--verify`를 다시 실행하고, 가능하면 배포 URL에서 Playwright smoke test를 수행한다.

## 요소 평가 기준

| 요소 | 통과 기준 |
|---|---|
| 선생님 시험 제작 | 세 시험의 제목·PDF·45문항·정답·배점이 교사 화면에서 일치 |
| 배포 | `테스트반` 학생만 열 수 있고 공개 게스트는 열 수 없음 |
| 학생 응시 | student3가 시험을 열고 OMR 답안을 제출 가능 |
| 필기 | PDF 위 필기 저장 후 교사·학생 리뷰에서 보관 상태 확인 가능 |
| 오답 리뷰 | 오답/미응답 필터, 정답 비교, 피드백 표시가 정확 |
| 재시험 | 원응시 오답만 출제되고 source attempt와 회복률이 연결 |
| 분석 | 교사 화면의 응시 수·점수·문항 오답 집계가 canonical 결과와 일치 |

각 항목은 `통과`, `부분 통과`, `실패`, `수동 확인 필요` 중 하나로 기록하고 근거가 되는 자동 테스트 또는 화면 경로를 함께 남긴다.

## 범위 제외

- 실제 학생이나 운영 반으로 배포
- 화법과 작문 선택과목 샘플
- 한국교육과정평가원 원본 또는 저작권 문구 수정
- 문제 해설 자동 생성
- 결제·요금제·카카오 발송 기능 변경
- 학생 인증 체계나 Supabase RLS 정책 변경
- 기존 대시보드 디자인 변경
