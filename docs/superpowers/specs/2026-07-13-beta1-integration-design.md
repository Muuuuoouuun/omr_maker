# Beta1 선별 통합 설계

## 문서 상태

- 작성일: 2026-07-13
- 작업 브랜치: `beta1`
- 기준 브랜치: `main` (`e0d05b9`)
- 통합 대상: `origin/premier0.1_cle` (`6fa225e`)
- 공통 조상: `c764681`
- 목적: `main`의 최신 기능을 보존하면서 `premier0.1_cle`의 서버 신뢰 경계와 학생·교사 개선을 검증 가능한 단위로 선별 통합한다.

## 1. 배경

`main`과 `premier0.1_cle`은 공통 조상 이후 각각 13개와 25개의 고유 커밋을 가진다. 대상 브랜치는 71개 파일에서 약 7,687줄을 추가하고 838줄을 제거하며, 자동 병합 시 11개 파일과 31개 충돌 구간이 발생한다.

대상 브랜치에는 정답 제거 solve payload, 서버 채점, 서명된 학생 세션, 학생 Q&A, 재시험 회복 분석 등 가져올 가치가 큰 기능이 있다. 반면 현재 상태에는 학생 사칭, 조직 격리 누락, 게스트 제출 FK 불일치, 비원자적 제출, 서버 경로 E2E 부재가 남아 있다. 따라서 직접 merge나 전체 cherry-pick은 `beta1`의 통합 방식으로 사용하지 않는다.

## 2. 목표

1. `main`의 반환 첨삭, 시험 입장 확인, 저장 복구, 다중 색상 필기 도구를 보존한다.
2. 학생용 시험 payload에서 정답·해설·PIN·정답 PDF를 제거한다.
3. 점수·소유자·조직·문항 결과를 서버에서 결정한다.
4. 서버가 검증한 학생·게스트 신원만 서비스롤 데이터 경계에 진입하도록 한다.
5. 모든 서비스롤 조회와 쓰기를 조직 및 실제 프로필에 결합한다.
6. 제출을 멱등적이고 원자적인 작업으로 만든다.
7. 실제 Supabase 경로를 포함한 회귀 테스트가 통합 결과를 증명하도록 한다.
8. 각 단계가 독립적으로 검토·테스트·되돌리기 가능하도록 커밋을 작게 유지한다.

## 3. 비목표

- `premier0.1_cle`의 대형 CSS 재설계를 한 번에 반영하지 않는다.
- 기존 화면 파일을 대상 브랜치 버전으로 통째로 교체하지 않는다.
- 결제·실사용량 서버 강제 등 별도 제품 범위를 추가하지 않는다.
- 교사 B1–B4 서버 이관이 끝나기 전에 production RLS를 활성화하지 않는다.
- 보안 경고만 추가하고 취약한 런타임 경로를 유지하는 방식은 완료로 인정하지 않는다.

## 4. 검토한 통합 방식

### A. `main` 기준 선별 통합, 채택

필요한 기반 커밋과 파일을 순서대로 가져오고, 충돌 화면은 `main`을 구조적 기준으로 수동 포팅한다. 작업량은 더 들지만 기능 손실과 보안 회귀를 단계별 테스트로 차단할 수 있다.

### B. 직접 merge 후 충돌 해결, 기각

Git 충돌은 한 번에 보이지만 의미상 충돌이 묻힌다. 특히 학생 리뷰와 교사 attempt에서 한쪽 파일을 선택하면 반환 첨삭 또는 Q&A 중 하나가 사라진다.

### C. 대상 브랜치 전체 순차 cherry-pick, 기각

25개 커밋이 patch-id 기준으로 모두 고유하지만 펜, 입장, 리뷰 기능은 `main`에 다른 형태로 존재한다. 순차 cherry-pick은 같은 충돌을 반복하고 의미상 중복까지 포함한다.

## 5. 통합 아키텍처

### 5.1 신뢰 경계

```text
학생 로그인 입력
  -> 서버 명단/시작코드 검증
  -> 조직·프로필에 결합된 서명 세션
  -> 학생 서버 액션
  -> 조직+소유자 조건의 서비스롤 조회
  -> 안전한 solve/review projection

답안 제출
  -> 런타임 입력 검증
  -> 서버 시험 조회 및 접근 검증
  -> 서버 채점
  -> 멱등성 키 확인
  -> attempt + question results 원자 저장
  -> 소유자 범위 결과 반환
```

학생 클라이언트는 Supabase에서 전체 시험 row를 직접 읽지 않는다. solve 화면에는 `SolvableExam`, 제출 후 본인 리뷰에는 `ReviewableExam`, 대시보드에는 정답이 없는 시험 요약만 전달한다.

### 5.2 신원 모델

서명 세션에 다음 서버 검증 값을 포함한다.

- `kind`: `student` 또는 `guest`
- `organizationId`
- `studentProfileId` 또는 서버 발급 `guestId`
- `studentId`
- 접근 가능한 `groupId`
- 표시용 이름과 지역 정보
- 발급·만료 시각

서버 액션은 임의 객체 spread로 신원을 구성하지 않는다. 허용 필드만 명시적으로 복사하며, 학생 세션은 서버 명단과 시작코드 검증에 성공한 경우에만 발급한다.

### 5.3 데이터 소유권

- attempt 조회: `organization_id + student_profile_id`, 게스트는 `organization_id + guest_id`
- 시험 조회: 조직, 배정, 그룹, PIN, 기간 조건을 서버에서 판정
- 리뷰 조회: 소유 attempt를 먼저 확인한 뒤 정답 포함 projection 반환
- Q&A 갱신: attempt 전체 JSON 덮어쓰기 대신 문항별 원자 갱신 또는 버전 검증 사용
- 제출: 클라이언트 idempotency key에 유일성 제약을 적용하고 단일 DB transaction/RPC로 저장

## 6. 단계별 통합

### 단계 0. 기준선 고정

- `main`의 unit, lint, build 결과를 기준선으로 기록한다.
- 반환 첨삭, 입장 확인, 다중 형광펜, 저장 복구를 회귀 테스트로 고정한다.
- 대상 브랜치의 안전한 projection 및 서버 채점 테스트를 독립적으로 가져온다.

### 단계 1. 서버 기반 모듈 선별 도입

다음 대상 커밋을 구현 참고 순서로 사용한다.

```text
3705d87 6805a15 ba28b3f ce232f4 efbdac3
7ed1f2f 481ba38 e0dc8cf cedeec0 a89c1c1
```

`studentServerSession`, `examSolvePayload`, `studentExamCore`, `supabaseServerAdmin`, 관련 테스트를 우선 도입한다. 커밋이 현재 타입과 충돌하면 커밋 전체를 강제로 적용하지 않고 해당 책임 단위만 포팅한다.

### 단계 2. 보안·데이터 블로커 해결

- 학생 세션 발급을 서버 명단/시작코드 검증과 결합한다.
- 세션과 모든 서비스롤 쿼리에 조직 및 프로필 식별자를 추가한다.
- 게스트·임시 학생의 `student_profile_id` 처리 규칙을 스키마와 일치시킨다.
- production에서 서비스롤 누락 시 로컬 정답 포함 폴백을 금지한다.
- 제출 멱등성 키와 원자 저장을 구현한다.
- PIN 제한을 공유 저장소의 원자 카운터로 바꾸고 시험 전체 고정 lockout을 제거한다.

이 단계가 끝나기 전에는 학생 서버 경계를 완료된 보안 기능으로 표시하지 않는다.

### 단계 3. `main` 화면에 서버 경로 배선

- `src/app/solve/[id]/page.tsx`: `main`의 입장 확인과 필기 UI를 유지하면서 서버 load/submit을 추가한다.
- `src/app/student/dashboard/page.tsx`: 미열람 피드백 표시를 유지하면서 서버 소유 attempt 및 안전한 시험 요약을 사용한다.
- `src/app/student/review/[attemptId]/page.tsx`: 반환 첨삭을 유지하면서 소유권 검증 리뷰와 학생 Q&A를 추가한다.
- `src/app/teacher/attempt/[attemptId]/page.tsx`: 첨삭 작성 기능을 유지하면서 Q&A 답변을 추가한다.
- `src/utils/storage.ts`: `main`의 검증·복구 로직을 유지하면서 게스트 login ID 보정만 추가한다.

### 단계 4. 독립 제품 기능 통합

- 계정별 교사 플랜 바인딩을 적용하되 미지정 계정은 항상 Free로 초기화한다.
- 재시험 회복 분석과 느린 정답 신호를 기존 프리미엄 분석에 추가한다.
- 학생 질문 인박스와 답변 상태를 대시보드에 추가한다.
- pending sync는 flush 중 새 항목을 잃지 않도록 저장소 병합 방식으로 수정한다.

### 단계 5. 필기·디자인 통합

펜/형광펜 개선은 별도 커밋 묶음으로 처리한다.

- `main`의 획 지우개, 다중 색상 형광펜, 커서 API 유지
- 대상 브랜치의 parse cache, 접근성 label, test id, UI 세부 개선만 포팅
- `c3692b9`의 학생 리뷰 재설계와 대형 CSS는 기능 통합 후 시각 QA를 거쳐 선택 적용

## 7. 충돌 해결 기준

다음 11개 파일은 자동 선택을 금지한다.

```text
e2e/teacher-pages.spec.ts
src/app/create/page.tsx
src/app/globals.css
src/app/page.tsx
src/app/solve/[id]/page.tsx
src/app/student/dashboard/page.tsx
src/app/student/review/[attemptId]/page.tsx
src/app/teacher/attempt/[attemptId]/page.tsx
src/components/PDFViewer.tsx
src/lib/drawingCursors.ts
src/utils/storage.ts
```

각 파일은 `main` 버전을 시작점으로 사용한다. 대상 기능은 함수, 상태, 타입, 스타일 블록 단위로 옮기며, 이동 직후 해당 기능의 회귀 테스트를 실행한다.

## 8. 오류 처리와 사용자 경험

- 인증 실패, 접근 거부, 시험 없음, 네트워크 실패를 서로 다른 상태로 표시한다.
- 명시적 `denied` 응답은 로컬 폴백으로 우회하지 않는다.
- 리뷰·교사 attempt·시험 상세는 영구 spinner 대신 재시도 가능한 오류 화면을 제공한다.
- 제출 중 버튼을 비활성화하고 저장 상태를 표시한다.
- 서버 제출 실패 시 답안 draft는 유지하되, 정답 없는 서버 시험은 클라이언트 채점하지 않는다.
- 타이머 복구 시 마지막 저장 이후 경과 시간을 차감하고 만료되면 자동 제출 경로로 진입한다.

## 9. 테스트 전략

### 단위 테스트

- 세션 필드 allowlist, 위조, 만료, 조직 결합
- solve/review payload에서 금지 필드 제거
- 서버 채점과 재시험 범위
- 조직+소유자 조회 조건
- 게스트/임시 학생 row 매핑
- 제출 idempotency와 부분 실패 롤백
- PIN 동시 요청 제한
- pending sync 동시 갱신
- 타이머 reload 경과 시간

### 통합 테스트

- 실제 FK가 있는 테스트 DB에서 학생·게스트 제출
- 두 조직에 동일 학생 ID가 있을 때 상호 격리
- attempt와 question results의 원자 저장
- 명시적 접근 거부 후 로컬 전체 시험을 읽지 않음
- 서비스롤 누락 production 환경의 fail-closed 동작

### E2E 테스트

- 서버 검증 학생 로그인과 서명 세션
- 정답 없는 solve payload
- PIN 성공·실패·제한
- 제출 후 서버 채점 결과
- 본인 리뷰 허용과 타인 리뷰 거부
- 반환 첨삭과 Q&A 동시 동작
- 입장 확인과 그룹 제한
- 태블릿 OMR 패널 기본 상태 및 펼치기 흐름
- 모바일·태블릿 제출
- 다중 색상 형광펜과 획 지우개

기존 E2E가 Supabase를 비활성화하므로 서버 경로 전용 테스트 프로젝트를 추가한다. CI는 설정된 Chromium과 WebKit 브라우저를 모두 설치해야 한다.

## 10. 출시 게이트

다음을 모두 만족해야 `beta1` 통합 완료로 판단한다.

1. `npm test`, `npm run lint`, `npm run build` 통과
2. 기본 Playwright 전체 통과 및 반복 실행에서 태블릿 flake 미발생
3. 실제 Supabase 테스트 환경의 서버 경로 E2E 통과
4. 학생 세션을 임의 `studentId`·`groupId`로 발급할 수 없음
5. 서비스롤 조회가 조직과 소유자에 모두 제한됨
6. 게스트·임시 학생 제출이 실제 FK 스키마에서 성공함
7. 중복 제출과 부분 저장이 재현되지 않음
8. `main`의 첨삭·입장·필기·저장 복구 회귀 없음
9. 교사 B1–B4 서버 이관과 production RLS 리허설 전까지 출시 차단 상태 유지

## 11. 롤백과 커밋 정책

- 각 단계는 독립 커밋으로 구성하며 기능·테스트·문서를 같은 커밋에 포함한다.
- 대형 화면과 CSS는 서버 경계 커밋과 분리한다.
- 단계별 실패 시 해당 단계 커밋만 revert할 수 있어야 한다.
- production RLS는 별도 운영 변경으로 취급하고 적용 전 스테이징 리허설과 롤백 SQL을 준비한다.

## 12. 완료 정의

`beta1`은 두 브랜치의 파일을 단순히 합친 상태가 아니다. `main`의 사용자 기능을 보존하고, 대상 브랜치의 서버 기능이 검증된 신원·테넌트·DB 계약·멱등성 경계 안에서 작동하며, 이를 unit·통합·E2E 테스트로 증명한 상태가 완료다.
