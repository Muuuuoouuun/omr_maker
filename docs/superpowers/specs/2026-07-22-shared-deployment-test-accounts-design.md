# 공유 배포 테스트 계정과 목업 데이터 격리 설계

## 목표

Vercel에 배포된 OMR Maker에서 관리자 1명, 강사 3명, 학생 3명이 로그인할 수 있게 한다. 관리자와 강사는 하나의 Supabase 테스트 워크스페이스에서 학생·시험·로스터 데이터를 공유한다. 일반 계정에는 자동 샘플 데이터를 표시하지 않고, 기존 분석 샘플은 공개 목업 계정에서만 유지한다.

## 계정 구성

| 구분 | 로그인 ID | 표시 이름 | 비밀번호 또는 시작 코드 | 역할 | 체험 요금제 상한 |
|---|---|---|---|---|---|
| 관리자 | `admin` | 관리자 | `admin1234` | `admin` | Academy |
| 강사 | `teacher1` | 강사 1 | `teacher1234` | `teacher` | Free |
| 강사 | `teacher2` | 강사 2 | `teacher1234` | `teacher` | Pro |
| 강사 | `teacher3` | 강사 3 | `teacher1234` | `teacher` | Academy |
| 학생 | `student1` | 학생 1 | `ABC234` | student | 해당 없음 |
| 학생 | `student2` | 학생 2 | `BCD345` | student | 해당 없음 |
| 학생 | `student3` | 학생 3 | `CDE456` | student | 해당 없음 |

간단한 비밀번호와 시작 코드는 사용자가 요청한 배포 테스트 전용 값이다. 실제 사용자나 민감한 학생 데이터를 저장하는 운영 계정으로 재사용하지 않는다. Vercel에는 교사 비밀번호 원문을 저장하지 않고 기존 PBKDF2 형식의 해시만 저장한다.

## 워크스페이스와 역할

네 교사 계정은 하나의 명시적 테스트 조직 ID를 사용한다. 로그인 자격증명의 `organizationId`를 서명된 교사 세션에 포함하고, 모든 교사 데이터 게이트웨이는 해당 조직 ID를 사용한다. 계정별 `teacherId`는 유지하여 감사 로그와 조직 회원 행에서 사용자를 구분한다.

Supabase 조직 회원 역할은 `admin` 계정을 `admin`, `teacher1`~`teacher3`을 `teacher`로 저장한다. 이번 범위는 회원 역할을 정확히 기록하고 세션에서 보존하는 것까지다. 기존 화면에 새로운 역할별 메뉴 제한을 추가하지 않는다.

## 요금제 모델

현재 서버 요금제는 조직 단위 권위값이다. 공유 테스트 조직의 요금제는 Academy로 저장한다. 교사별 차등 체험을 위해 자격증명의 `plan`을 계정별 **요금제 상한**으로 해석한다.

서버가 계산하는 유효 요금제는 조직 요금제와 계정 상한 중 낮은 등급이다.

- `admin`: min(Academy, Academy) = Academy
- `teacher1`: min(Academy, Free) = Free
- `teacher2`: min(Academy, Pro) = Pro
- `teacher3`: min(Academy, Academy) = Academy

계정 상한은 조직이 보유하지 않은 권한을 높일 수 없고 낮출 수만 있다. 유효 요금제 계산은 서명된 서버 세션을 기준으로 하며 브라우저 저장값을 권한 판정에 사용하지 않는다. 사용량은 기존과 동일하게 공유 조직 단위로 집계한다.

## 목업 데이터 격리

분석용 `demoData`는 `omr-showcase` 목업 교사 세션에서만 활성화한다. `NODE_ENV !== production`이나 실제 데이터가 비어 있다는 이유만으로 일반 관리자·강사 화면에 샘플을 넣지 않는다.

다음 화면을 동일한 원칙으로 정리한다.

- 교사 대시보드: 일반 계정은 데이터가 없으면 실제 빈 상태 표시
- 학생/그룹 관리: 일반 계정은 빈 명단 상태 표시
- 실시간 모니터링: 일반 계정은 가상 시험·학생을 생성하지 않음
- 공개 목업 계정: 기존 샘플 시험·학생·분석 데이터 유지

샘플 데이터는 표시 전용이며 공용 테스트 조직의 Supabase 테이블에 저장하지 않는다.

## Supabase 테스트 데이터

공유 조직에 다음 최소 데이터를 멱등적으로 등록한다.

- Academy 요금제의 테스트 조직 1개
- 조직 회원 4명과 교사 프로필 4개
- `테스트반` 1개
- 학생 프로필 3개
- 학생과 테스트반의 활성 등록 관계 3개
- 학생 시작 코드 검증용 해시 메타데이터 3개

학생 시작 코드 해시는 기존 `studentAccessCode` 규약과 Vercel의 학생 세션 비밀값을 사용해 생성한다. 학생 로그인은 배포된 앱에서 테스트 조직 링크, 이름, 테스트반, 학생 ID, 시작 코드 조합으로 검증한다. 데이터 등록 스크립트는 같은 ID로 다시 실행해도 중복 행을 만들지 않는 upsert 방식으로 작성한다.

## Vercel 설정

연결된 `omr-maker` 프로젝트의 Preview와 Production 환경에 다음 서버 설정을 적용한다.

- 네 개 교사 계정의 ID, 표시 이름, 이메일, 비밀번호 해시, 조직 ID, 회원 역할, 요금제 상한을 담은 `TEACHER_ACCOUNTS`
- 강한 임의값의 `TEACHER_SESSION_SECRET`
- 학생 코드와 세션 서명에 필요한 `STUDENT_SESSION_SECRET`
- 기존 Supabase URL, 공개 키, 서비스 역할 키는 유지하고 누락 여부만 확인

환경변수 변경 후 재배포하여 새 서버 인스턴스가 값을 읽게 한다. 비밀값과 비밀번호 해시는 저장소에 커밋하지 않는다.

## 오류 처리

- 자격증명에 유효하지 않은 조직 ID, 역할 또는 요금제가 있으면 해당 계정을 무시하고 배포 준비 상태에 구성 오류를 표시한다.
- 세션의 명시적 조직 ID가 없으면 기존 계정별 `teacher_<hash>` 워크스페이스 계산을 유지하여 하위 호환한다.
- Supabase 조직 요금제를 읽지 못하면 기존 fail-closed 정책을 유지한다.
- 계정 상한이 없으면 조직 요금제를 사용한다. 알 수 없는 상한 문자열이 있으면 해당 자격증명을 파싱 단계에서 거부한다.
- 배포 도구 인증이나 Supabase 서비스 역할 접근이 없으면 코드 변경과 검증 결과를 보존하고, 외부 등록은 완료로 표시하지 않는다.

## 검증

자동 테스트는 다음을 확인한다.

- 자격증명 JSON에서 조직 ID, 회원 역할, 요금제 상한을 파싱한다.
- 서명된 교사 세션이 새 필드를 보존하고 변조된 쿠키를 거부한다.
- 명시적 조직 ID가 네 계정에 같은 워크스페이스를 제공하면서 actor ID는 서로 다르게 유지된다.
- 조직 회원 bootstrap이 관리자와 강사 역할을 구분한다.
- 유효 요금제가 Academy/Free/Pro/Academy로 계산되며 상한이 조직 플랜을 올리지 못한다.
- 일반 개발 계정의 빈 대시보드·명단·실시간 화면에 목업 데이터가 나타나지 않는다.
- `omr-showcase` 계정에는 기존 목업 데이터가 계속 나타난다.
- Supabase 시드가 학생 3명과 시작 코드 레코드를 멱등적으로 만든다.

배포 후에는 Preview와 Production에서 다음 smoke test를 수행한다.

1. `admin`, `teacher1`, `teacher2`, `teacher3` 로그인
2. 네 계정이 동일한 테스트반과 학생 3명을 조회하는지 확인
3. 각 교사 계정의 유효 요금제가 계획한 등급인지 확인
4. `student1`~`student3`이 각 시작 코드로 로그인하는지 확인
5. 일반 계정에 분석 샘플이 없고 공개 목업 계정에는 남아 있는지 확인

## 범위 제외

- Supabase Auth 기반 교사·학생 계정 전환
- 일반 사용자용 회원가입과 비밀번호 재설정
- 역할별 메뉴와 작업 권한의 전면 RBAC 구현
- 회원별 결제 또는 회원별 독립 사용량 집계
- 목업 데이터를 공용 테스트 조직의 실제 데이터로 복사하는 기능
