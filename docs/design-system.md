# 디자인 시스템

이 문서는 `src/app/globals.css`의 `:root` 토큰과 그 사용 규칙을 정리합니다. 목적은 하나입니다 —
**새 화면을 만들 때 값을 새로 고르지 말고, 아래 토큰에서 고르게 하는 것.** `globals.css`가
최초 리디자인(1,108줄) 이후 7,600줄대로 늘어난 가장 큰 원인이 인라인 스타일로 그때그때 값을
정한 것이었고, 이 문서가 없으면 같은 드리프트가 반복됩니다.

배경과 진단은 [디자인 디벨롭 기획안](.)을 참고하세요 (2026-07-22, 팀 아티팩트).

## 원칙

1. **구조보다 디테일.** 레이아웃/정보구조를 바꾸는 리팩터링과, 타이포그래피·컬러·모션 같은
   디테일 다듬기는 별개 작업입니다. 이 문서는 후자만 다룹니다.
2. **토큰에서 고른다.** `0.86rem`, `fontWeight: 950` 같은 손으로 고른 값 대신 아래 스케일에서
   가장 가까운 값을 씁니다. 스케일에 맞는 값이 없으면 — 정말 필요할 때만 — 스케일에 새 단계를
   추가하고 문서를 함께 업데이트합니다. 컴포넌트마다 조용히 새 값을 만들지 않습니다.
3. **의미 있는 색만 분리한다.** `--error`(시스템 오류)와 `--grade-red`(채점/정답 관련)처럼,
   같은 "빨강"이라도 사용자가 다른 의미로 읽어야 하면 별도 토큰을 씁니다. 반대로 의미가 같으면
   토큰을 늘리지 않습니다.
4. **하나의 절제된 시그니처.** OMR 답안지 마킹 모티프처럼 이 제품만의 단서는 힘이 셉니다 —
   그래서 여러 곳에 쓰면 장식이 되어버립니다. 화면당 한 곳, 정말 중요한 지점에만 씁니다.

## 컬러

| 토큰 | 라이트 | 다크 | 용도 |
|---|---|---|---|
| `--primary` | `#4f46e5` | `#818cf8` | 브랜드 액센트, 주요 액션 |
| `--secondary` | `#ec4899` | — | 보조 액센트 (드물게 사용) |
| `--accent` | `#8b5cf6` | — | 보조 액센트 (드물게 사용) |
| `--success` | `#10b981` | — | 성공, 증가 추세, 완료 상태 |
| `--warning` | `#f59e0b` | — | 주의, 확인 필요 |
| `--error` | `#ef4444` | — | 시스템 오류, 실패, 감소 추세 |
| `--grade-red` | `#c02b3c` | `#f0687a` | **채점/정답 관련 강조 전용.** 오답 수, 채점 마킹 등. `--error`와 절대 겹쳐 쓰지 않습니다 — "틀렸다"가 "고장났다"로 읽히면 안 됩니다. |
| `--foreground` / `--muted` / `--border` | — | — | 텍스트/보더 뉴트럴, 항상 이 토큰으로 |

각 색은 `-soft`(배경, 8~12% 불투명도)와 `-line`(보더, 22~32% 불투명도) 변형이 있습니다
(`--primary-soft`, `--grade-red-line` 등). 새 상태 배지를 만들 때 직접 `rgba()`를 쓰지 말고
이 패턴을 따르세요.

`--error`를 쓸지 `--grade-red`를 쓸지 헷갈리면: **"이 숫자가 틀려서 나쁜 건가, 시스템이 문제라서
나쁜 건가?"** 오답 수·정답률처럼 시험 내용에 대한 것이면 `--grade-red`, 동기화 실패·저장 오류처럼
시스템 상태에 대한 것이면 `--error`.

## 타이포그래피

### 폰트 페어링

- **한글 제목·본문 → Pretendard** (`next/font/local`로 셀프호스팅, 가변 폰트). 이 앱 텍스트의
  대부분(한글)은 예전에 Geist가 커버하지 못해 OS 기본 폰트로 대체 렌더링되고 있었습니다.
  `body`가 이미 Pretendard 우선 스택으로 설정되어 있어서, **대부분의 새 텍스트는 폰트를 직접
  지정할 필요가 없습니다** — 상속만으로 충분합니다.
- **숫자·통계·mono 데이터 → Geist**. 점수·정답률처럼 정렬이 중요한 숫자는 `.numeric-emphasis`
  클래스를 쓰면 Geist 우선 스택 + `tabular-nums`가 자동 적용됩니다. 새 통계 수치를 만들 때
  폰트를 직접 지정하지 말고 이 클래스를 재사용하세요.
- 두 폰트를 섞어 쓰는 이유: 이 제품은 숫자(점수·통계)가 핵심 콘텐츠라, "문장은 Pretendard,
  숫자는 Geist"가 임의 조합이 아니라 역할 분담입니다.
- **주의 — `--font-sans` 같은 합성 커스텀 프로퍼티를 새로 만들지 마세요.** `--font-pretendard`
  / `--font-geist-sans`는 Next.js 폰트 로더가 `<body>` className에 스코프해서 주입합니다.
  `:root`에 `--font-sans: var(--font-pretendard), var(--font-geist-sans), ...`처럼 이 값들을
  다시 조합하는 변수를 선언하면, `<body>`보다 위(예: `:root`)에서 그 변수를 참조할 때
  `var()` 체인이 invalid로 취급되어 `font-family` 선언 전체가 조용히 무효화됩니다(전체
  본문이 브라우저 기본 serif로 렌더링됨 — 실제로 겪은 버그입니다). `--font-pretendard` /
  `--font-geist-sans`를 직접, 사용하는 지점에서 참조하세요 (`body`, `.numeric-emphasis`가
  이미 이 패턴입니다).

### 자간(letter-spacing)

라틴 디스플레이 타이포그래피의 강한 음수 자간(`-0.02em` ~ `-0.045em`) 관행을 한글에 그대로
쓰지 마세요 — 한글은 자간을 좁히면 답답해 보입니다. 한글이 포함된 헤딩/타이틀은 `-0.01em`
이내로 유지하거나 자간을 아예 지정하지 않습니다. 강한 음수 자간은 라틴 문자·숫자로만 이루어진
요소(로고 워드마크, 숫자 강조)에 한정하세요.

### 타입 스케일

| 토큰 | 크기 | 용도 |
|---|---|---|
| `--type-micro` | `0.7rem` | 최소 보조 텍스트 (배지 안 캡션 등) |
| `--type-caption` | `0.76rem` | 캡션, 보조 설명 |
| `--type-label` | `0.84rem` | 버튼/탭 라벨, 폼 라벨 |
| `--type-body-sm` | `0.88rem` | 축약 본문 |
| `--type-heading-sm` | `1.02rem` | 카드/섹션 소제목 |
| `--type-heading-md` | `1.1rem` | 페이지 내 섹션 h3 (카드 제목보다 한 단계 큼) — `--type-heading-sm`과 `--type-heading-lg` 사이가 비어 있어서 다들 `1.1~1.35rem`을 손으로 흩어 쓰고 있었습니다 |
| `--type-heading-lg` | `clamp(1.8rem, 3vw, 2.45rem)` | 페이지 타이틀 |
| `--type-metric` | `clamp(1.6rem, 2.2vw, 2.05rem)` | 큰 숫자 강조 (통계 카드 값) |

새 텍스트 크기가 필요하면 위 표에서 가장 가까운 값을 먼저 고르세요. `0.82rem`을 쓰고 싶다면
대개 `--type-label`(0.84rem)이나 `--type-caption`(0.76rem) 중 의도에 더 맞는 쪽입니다.

## 스페이싱 · 라운드 · 섀도우

라운드(`--radius-sm` 0.5rem / `-md` 0.75rem / `-lg` 1rem / `-xl` 1.5rem / `-full` 9999px)와
섀도우(`--shadow-sm/md/lg/xl`, `--shadow-glass`, `--shadow-glow`)는 이미 있는 스케일을 그대로
씁니다. 카드는 기본적으로 `--radius-lg`, 배지/필은 `--radius-full`, 버튼/인풋은 `--radius-md`가
기존 관행입니다.

## 모션

| 토큰 | 지속시간 | 이징 | 용도 |
|---|---|---|---|
| `--transition-fast` | 0.15s | `cubic-bezier(0.4,0,0.2,1)` | 호버, 포커스 |
| `--transition-base` | 0.3s | `cubic-bezier(0.4,0,0.2,1)` | 일반 상태 전환 |
| `--transition-smooth` | 0.5s | `cubic-bezier(0.25,1,0.5,1)` | 큰 레이아웃 변화 |
| `--transition-spring` | 0.4s | `cubic-bezier(0.175,0.885,0.32,1.275)` | 등장/강조 애니메이션 (통통 튀는 느낌) |
| `--ease-spring` | (듀레이션은 호출부에서) | `cubic-bezier(0.34,1.56,0.64,1)` | 카드/KPI 등장 팝 |
| `--ease-spring-soft` | — | `cubic-bezier(0.34,1.2,0.5,1)` | 바 차트 성장처럼 덜 튀는 등장 |
| `--ease-out-soft` | — | `cubic-bezier(0.22,1,0.36,1)` | 부드러운 정착(settle), 탭 전환 |

두 스케일이 존재하는 이유: `--transition-*`는 호버/포커스 같은 짧은 상태 전환용이고,
`--ease-*`는 차트/KPI 등장 같은 "리빌" 애니메이션 전용입니다. 새 모션을 만들 때 이 중 어떤 걸
써야 할지 헷갈리면 — 사용자 액션에 대한 즉각 반응이면 `--transition-*`, 화면에 나타나는
연출이면 `--ease-*`.

새 모션(카운트업, 리빌 애니메이션 등)을 만들 때 duration/easing을 직접 정하지 말고 이 스케일에서
고르세요. 컴포넌트마다 타이밍이 미묘하게 다르면 "이것저것 붙여넣은" 느낌이 나고, 하나의 스케일을
공유하면 그것만으로 "조율된 느낌"이 생깁니다. `prefers-reduced-motion`과 앱 자체의
`html[data-motion="off"]`(설정에서 모션 끄기) 둘 다 존중해야 합니다 — 기존 `.numeric-emphasis`
애니메이션이 그 패턴의 예시입니다.

**Web Animations API(`element.animate()`)에서는 `var(--ease-*)`가 작동하지 않습니다** — WAAPI의
`easing` 옵션은 리터럴 CSS 이징 문자열만 받고 커스텀 프로퍼티를 해석하지 않습니다. JS에서 앱
이징을 재사용하려면 토큰의 실제 `cubic-bezier(...)` 값을 리터럴로 복사하되, 주석으로
"`--ease-spring`과 동기화 — globals.css에서 값이 바뀌면 여기도 같이 바꿀 것"처럼 출처를
남기세요. 그래야 나중에 토큰만 고치고 JS 복사본을 깜빡하는 일이 생겨도 코드에서 바로 보입니다.

동일한 목적(순차 등장 stagger)에 여러 파일이 각자 다른 값(40/42/60/65/80ms)을 쓰고 있는
사례가 발견됐습니다 — 새로 stagger를 추가한다면 이 중 하나(대략 60ms대)에 맞추는 것이
새 값을 또 만드는 것보다 낫습니다.

## 컴포넌트 재사용

- **상태 pill/배지** (동기화 상태, 데이터 상태, 오답 수 등): `src/components/dashboard/StatusPill.tsx`
  공유 컴포넌트를 씁니다. 화면마다 `style={{ borderRadius: 'var(--radius-full)', border: ..., background: ... }}`
  를 새로 쓰지 마세요 — 패딩·아이콘 비율·폰트 크기가 화면마다 미묘하게 달라지는 원인이었습니다.
- **통계 숫자 강조**: `.numeric-emphasis` 클래스 (폰트·tabular-nums·등장 애니메이션 포함).
- 새 공유 컴포넌트가 필요하면 `src/components/dashboard/`(대시보드 전용) 또는
  `src/components/`(범용) 중 실제 재사용 범위에 맞는 곳에 둡니다. `ui/` 같은 별도 하위 폴더
  관행은 아직 없습니다 — 새로 만들지 마세요.

## 새 UI를 추가할 때 체크리스트

1. 색이 필요하면 위 컬러 토큰에서 고른다. 새 rgba를 손으로 만들지 않는다.
2. 폰트 크기는 `--type-*`에서 고른다. 숫자 강조는 `.numeric-emphasis`를 재사용한다.
3. 한글 헤딩에 강한 음수 자간을 넣지 않는다.
4. 상태 배지가 필요하면 `StatusPill`을 재사용한다. 없는 변형이 필요하면 컴포넌트를 확장하지,
   옆에 새로 인라인으로 만들지 않는다.
5. 모션은 `--transition-*`에서 고른다.
6. 장식적 모티프(답안지 텍스처 등)는 화면당 한 곳 이상 쓰지 않는다.
